# セミナー分析 成果物リスト

製造業ITツール批判的検証フレームワークに基づき、セミナーの文字起こしから作成すべき成果物を定義する。

**前提:** 手元にある唯一の材料は「セミナーの文字起こし」のみ。すべての成果物はここから段階的に派生する。

---

## Phase 0: 【準備】文字起こしの構造化

### 成果物0-1: セミナー内容サマリー

**最終用途:** 以降のすべての分析作業の索引・目次として機能

**目的:** 文字起こしを構造化し、必要な情報への高速アクセスを可能にする

**作成に必要な材料:**
- セミナー文字起こし(原文)のみ

**記載項目例:**
- セミナー基本情報(日時、主催者、製品名)
- 章立て構成(タイムスタンプ付き)
  - 製品概要説明(○分〜○分)
  - 機能デモ(○分〜○分)
  - 導入事例紹介(○分〜○分)
  - 料金プラン説明(○分〜○分)
  - 質疑応答(○分〜○分)
- 発言者分類(ベンダー/参加者)
- キーワード索引(コスト、時間、削減、課題、連携等の出現箇所)
- 数値情報リスト(金額、時間、削減率、台数等)

---

## Phase 1: 【Impact】課題解決インパクトの検証

### 成果物1-1: ベンダー主張の課題リスト

**最終用途:** 成果物1-2(自社適合性チェック)の入力材料

**目的:** セミナーでベンダーが提示した「解決すべき課題」を網羅的に抽出する

**作成に必要な材料:**
- 成果物0-1: セミナー内容サマリー(特に課題説明部分の索引)

**記載項目:**
- 課題番号
- 課題の内容(ベンダーの表現のまま原文引用)
- 課題の種類(入力ミス、情報断絶、作業遅延、属人化等)
- 言及された業務シーン(現場入力、事務処理、データ集計等)
- 文字起こし該当箇所(ページ/時間)

**抽出キーワード:**
- 「課題」「問題」「困っている」「非効率」「手間」
- 「なぜ」「原因」「理由」等の因果関係の説明
- ネガティブ表現(「できない」「遅い」「ミスが多い」等)

---

### 成果物1-2: 自社適合性チェックシート

**最終用途:** 現場ヒアリングの質問票、経営層への説明資料

**目的:** ベンダーが主張する課題が自社に当てはまるか、真因か現象かを判定する

**作成に必要な材料:**
- 成果物1-1: ベンダー主張の課題リスト

**記載項目:**
- 課題番号(1-1と紐付け)
- ベンダー主張の課題(引用)
- 自社での該当業務の有無(有 / 無 / 類似)
- 自社での課題の実在性(未確認 / 要ヒアリング / 確認済み)
- 真因・現象の推定(真因 / 現象 / 不明)
- IT以外の解決策の可能性(ルール変更、レイアウト変更、廃止等)
- 確認が必要な情報(ヒアリング項目、調査内容)

**この段階での判定:**
- 文字起こしからの推定のみで記載
- 「未確認」「要ヒアリング」を積極的に使用
- 確実に言えることだけを断定する

---

### 成果物1-3: 機能一覧とECRS評価マトリクス

**最終用途:** 導入機能の優先順位付け、段階的導入計画の策定、費用対効果説明資料

**目的:** ツールの各機能をECRS原則で格付けし、価値の高い機能を特定する

**作成に必要な材料:**
- 成果物0-1: セミナー内容サマリー(特に機能説明部分の索引)
- 成果物1-1: ベンダー主張の課題リスト(機能との対応関係)

**記載項目:**
- ツール名/製品名
- 機能番号
- 機能名(ベンダーの表現のまま)
- 機能の説明(文字起こしから引用)
- 対応する課題(1-1の課題番号)
- ビフォー状態(ベンダー説明より)
- アフター状態(ベンダー説明より)
- ECRS分類(E / C / R / S / 不明)
- 分類の根拠(どの作業がどう変わるか)
- 価値評価(高 / 中 / 低 / 判定保留)
- 代替手段の可能性(紙、Excel、業務廃止、既存システム改修等)
- 不明点・確認事項(ベンダーへの追加質問)

**抽出キーワード:**
- 「自動化」「連携」「入力」「削減」「排除」等の動詞
- 「○○から○○へ」のビフォー・アフター表現
- 「不要になる」「なくなる」等の排除表現

---

### 成果物1-4: ベンダー主張の効果数値リスト

**最終用途:** 成果物1-5(自社効果試算)の参考値、効果の信憑性評価

**目的:** セミナーで提示された効果数値を網羅的に抽出し、前提条件を明確化する

**作成に必要な材料:**
- 成果物0-1: セミナー内容サマリー(特に導入事例部分と数値情報リスト)
- 成果物1-3: 機能一覧(効果と機能の紐付け)

**記載項目:**
- 事例番号
- 導入企業の属性(業種、規模、地域等 ※言及された範囲のみ)
- 対象業務
- 導入機能(1-3の機能番号)
- ビフォー数値(作業時間、人数、ミス件数等)
- アフター数値
- 削減効果(数値・%)
- 効果の表現(「○○が○○%削減」の原文)
- 前提条件・制約(言及された範囲)
- 効果発現までの期間
- 自社との類似度(高 / 中 / 低 / 不明)
- 信憑性評価(具体的 / 曖昧 / 誇大表現の疑い)

**抽出キーワード:**
- 「削減」「短縮」「○時間」「○分」「○%」
- 「導入前」「導入後」「従来」「改善」
- 作業頻度(「日次」「月次」「毎回」等)

---

### 成果物1-5: 自社効果試算シート(暫定版)

**最終用途:** ROI計算の基礎データ、稟議書の投資効果欄、現場へのメリット説明

**目的:** 自社に当てはめた場合の効果を試算する(現時点で入手可能な情報のみで)

**作成に必要な材料:**
- 成果物1-2: 自社適合性チェックシート(該当業務の有無)
- 成果物1-3: 機能一覧とECRS評価(価値の高い機能)
- 成果物1-4: ベンダー主張の効果数値リスト(参考値)

**記載項目:**
- 対象業務名
- 対応する機能(1-3の機能番号)
- ベンダー事例の効果(1-4からの参考値)
- 自社での推定効果(保守的・楽観的の2パターン)
  - 作業時間削減(人時/月、人時/年)
  - ミス削減(件数/月)
  - 金額換算(試算根拠を明記)
- 前提条件(「○○が実現した場合」)
- 効果の確実性(高 / 中 / 低 / 不明)
- 追加調査が必要な項目(現状作業時間、作業頻度、人件費単価等)

**この段階での制約:**
- 自社の実績値がないため「推定」「仮定」を明記
- ベンダー事例を参考に、保守的に見積もる
- 「要実測」「要ヒアリング」を積極的に記載

## Phase 1.5: 【客観性検証】ベンダー主張の裏付け調査

### 成果物1.5-1: ベンダー主張の信憑性検証シート

**最終用途:** 成果物4-1(総合評価レポート)の信頼性向上、経営層への客観的根拠提示

**目的:** ベンダーの主張を第三者情報で裏付け、誇大広告や片面的説明を見抜く

**作成に必要な材料:**
- 成果物1-1: ベンダー主張の課題リスト
- 成果物1-3: 機能一覧とECRS評価
- 成果物1-4: ベンダー主張の効果数値リスト
- インターネット調査結果

**記載項目:**

**1. 製品・ベンダーの評判調査**
- IT製品レビューサイトでの評価
  - ITreview、Gartner等での総合評価
  - 評価件数・最新レビュー日
  - 高評価項目 / 低評価項目
- ユーザーの生の声(特に低評価レビュー)
  - 不満点の傾向(機能、サポート、コスト等)
  - 解約理由・後悔している点
  - 「期待外れ」だった点
- ベンダーの事業継続性
  - 企業規模・財務状況(公開情報範囲)
  - 事業年数・顧客数の推移
  - 倒産リスクの評価

**2. 効果数値の妥当性検証**
- 導入事例企業の公開情報との照合
  - IR資料・プレスリリースでの言及有無
  - セミナー説明との一致度
  - 「盛っている」可能性の評価
- 同業他社の類似ツール導入効果
  - 業界標準の削減率
  - セミナー事例の妥当性(高すぎないか)
- 効果測定方法の妥当性
  - 計算式の透明性
  - 恣意的な前提条件の有無

**3. 隠された制約・トラブル事例**
- 導入失敗事例の有無
  - ニュース記事、ブログ、SNS
  - 失敗の要因(技術、コスト、定着等)
- バージョンアップ時の問題
  - 過去の不具合履歴
  - ユーザーの混乱・クレーム
- サポート対応への評判
  - Twitter検索、掲示板等
  - 対応速度・品質の評価
- 契約トラブル・訴訟リスク
  - 法的紛争の有無
  - 契約条件への不満

**4. 技術的な陳腐化リスク**
- 採用技術の新旧
  - レガシー技術への依存度
  - 最新技術トレンドへの対応
- 競合製品との技術的差異
  - 独自技術 vs 汎用技術
  - 技術的優位性の持続性
- 5年後の生存可能性
  - 市場トレンドとの適合性
  - 技術革新への追従力

**調査方法:**
- IT製品レビューサイト検索(ITreview、Capterra、G2等)
- Twitter/X、Reddit、はてなブックマーク等のSNS検索
- Googleニュース検索(製品名、ベンダー名)
- 導入事例企業のIR資料・プレスリリース検索
- 技術ブログ、Qiita、Stack Overflow等

### 成果物1.5-2: 競合製品比較マトリクス

**最終用途:** 製品選定の妥当性検証、経営層への代替案提示、価格交渉の材料

**目的:** ベンダー提示の「唯一性」を検証し、代替選択肢を把握する

**作成に必要な材料:**
- 成果物1-3: 機能一覧とECRS評価
- 成果物3-1: ベンダー提示コスト情報抽出シート
- インターネット調査結果

**記載項目:**

**1. 競合製品のリストアップ**
- 製品名 / ベンダー名
- カテゴリ(同一 / 類似 / 代替可能)
- 市場シェア・導入実績
- 日本国内での展開状況

**2. 機能比較マトリクス**
- ECRS評価の横比較
  - E(排除)機能の有無
  - C(結合)機能の有無
  - 各社の強み・弱み
- 独自機能 vs 業界標準機能
  - セミナー製品の差別化要因
  - 「普通の機能」を特別視していないか

**3. 価格帯の比較**
- 初期費用の比較
- 月額/年額の比較
- ユーザー単価の比較
- 価格の妥当性評価(高 / 適正 / 安)

**4. ユーザー評価の比較**
- レビューサイトでの評価点
- 各社の強み(レビューから)
- 各社の弱み(レビューから)

**5. シェア・トレンド分析**
- 市場シェアの推移
- ユーザー数の増減トレンド
- 業界での位置づけ(主流 / ニッチ / 衰退期)

**調査方法:**
- Google検索「(製品カテゴリ) 比較」「(製品名) 競合」
- IT製品比較サイト(ボクシル、アスピック等)
- SaaS比較サイト(海外: G2、Capterra等)
- 市場調査レポート(無料公開部分)

### 成果物1.5-3: 「やらない」選択肢の再検証

**最終用途:** ECRS の E(排除)の徹底検証、経営層への根本的な問題提起

**目的:** ツール導入以外の解決策をインターネット情報から発見する

**作成に必要な材料:**
- 成果物1-1: ベンダー主張の課題リスト
- 成果物1-2: 自社適合性チェックシート
- インターネット調査結果

**記載項目:**

**1. 業務廃止事例の収集**
- 課題番号(1-1と紐付け)
- 同じ課題を「やめる」ことで解決した事例
  - 企業名・業種
  - 廃止した業務内容
  - 廃止後の影響(問題なし / 代替策で対応)
- 規制緩和・ルール変更で不要になった業務
  - 法改正、業界ルール変更
  - デジタル化で様式変更(印鑑廃止等)

**2. 無料・低コスト代替手段**
- オープンソースツール
  - 製品名・機能概要
  - 導入事例・評判
  - 自社適合性(推定)
- Excel/Google Sheetsのテンプレート
  - 公開されているテンプレート
  - マクロ・関数での自動化可能性
- RPA(既存ツール)での自動化
  - 既存RPAツールでの実現可能性
  - 開発工数の概算

**3. 業界ベストプラクティス**
- 先進企業の業務効率化事例
  - 事例企業名・業種
  - 実施した施策
  - IT投資せずに実現した改善
- 「そもそもその業務は必要か?」の再定義
  - 業務目的の本質的な問い直し
  - 顧客価値に直結しない業務の特定

**調査方法:**
- Google検索「(業務名) 廃止 事例」「(業務名) 不要」
- 経営・業務改善系メディア(日経BP、ダイヤモンド等)
- DX事例サイト(経済産業省DX推進等)
- GitHub等でのオープンソース検索
- Excel/Googleスプレッドシートテンプレートサイト

## Phase 2: 【Retention】現場定着率の検証

### 成果物2-1: UI/UX要件抽出シート

**最終用途:** 成果物2-2(自社環境適合性評価)の入力材料、ベンダーへの追加質問リスト

**目的:** セミナーで説明されたUI/UXの特徴を抽出し、操作性を評価する材料を整理する

**作成に必要な材料:**

- 懸念事項(推定)
- 確認が必要な項目(実機テスト、現場視察等)

**この段階での制約:**
- 自社の実環境データがないため「推定」を明記
- 「要実機確認」「要現場視察」を積極的に記載
- 判断保留項目を明確化

- 導入事例での連携実績
- 質疑応答で言及された連携の課題

- 「連携」「API」「CSV」「インポート」「エクスポート」
- 「ERP」「生産管理」「会計」等のシステム名
- 「自動」「手動」「リアルタイム」「同期」

- Cost評価(A / B / C / D)

**5. 総合判定(暫定)**

- **【GO(条件付き)】** の場合:
  - 現時点での推奨理由
  - 期待効果(シナリオ別)
  - GO判定の前提条件(「○○が確認できれば」)
  - 本格検討前に必要な追加調査
  - 次のステップ(成果物4-2へ)
  
- **【WAIT(追加調査推奨)】** の場合:
  - 判断保留の理由
  - 潜在的なメリット
  - 潜在的なリスク
  - 判断に必要な追加情報(優先度付き)
  - 追加調査の実施方法
  - 再評価のタイミング
  
- **【NO(現時点で不適合)】** の場合:
  - 不適合の理由
  - 克服不可能な障壁
  - 代替案の方向性(業務廃止、既存ツール活用等)
  - 将来再検討する条件(技術進化、コスト低下等)

**6. 判定の信頼度評価**
- 高確度の判断材料(文字起こしから明確に読み取れた事項)
- 低確度の判断材料(推定・仮定に基づく事項)
- 判断精度への影響度(どの不確実性が最も判定を左右するか)

**7. 次のアクション(優先度順)**
- 最優先の追加調査項目
- ベンダーへの質問リスト
- 社内ヒアリング項目
- 実機デモ・トライアルの要否
- 競合製品の調査要否

---

### 成果物4-2: 追加調査計画書(WAIT判定時)

**最終用途:** 追加調査の実施指示、関係部署への協力依頼、調査予算の申請

**目的:** 判断に必要な追加情報を取得するための具体的な計画を立てる

**作成に必要な材料:**
- 成果物4-1: 総合評価レポート(特に「次のアクション」セクション)

**記載項目:**
- **調査項目リスト(優先度順)**
  1. ベンダーへの追加質問
     - 質問内容
     - 回答期限
  2. 社内ヒアリング
     - 対象者(部署・役職)
     - 質問項目
     - 実施時期
  3. 現場調査
     - 調査内容(作業時間測定、環境確認等)
     - 対象範囲
     - 実施時期
  4. 実機デモ・トライアル
     - 確認項目
     - 参加者
     - 実施時期
  5. 競合製品調査
     - 調査対象製品
     - 比較項目
     - 実施時期

- **調査スケジュール**
  - 各調査の開始・完了予定日
  - 全体の完了目標日

- **次回判定のタイミング**
  - 再評価実施日
  - 最終判定期限

---

### 成果物4-3: ベンダー回答書(NO判定時)

**最終用途:** ベンダーへの不採用通知、社内関係者への共有、将来の参照資料

**目的:** 不採用の理由を明確かつ丁寧に伝え、将来の関係を維持する

**作成に必要な材料:**
- 成果物4-1: 総合評価レポート(NO判定版)

**記載項目:**
- セミナー参加への感謝
- 不採用の理由(具体的だが配慮ある表現)
- 評価したポイント(ポジティブな側面)
- 将来再検討する条件(あれば)
- 今後の関係性(情報提供の希望等)
