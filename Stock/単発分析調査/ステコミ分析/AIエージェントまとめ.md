# 【打合せ資料骨子案】生産領域におけるAIエージェント活用の検討

## 0. AIエージェントとは(業界標準の定義)

### 定義

AIエージェント：**LLM(大規模言語モデル)が自律的にツールを使い、ループ(繰り返し)の中で目標を達成しようとするシステム**です。  
事前に定義されたコードパス(手順)に従う「ワークフロー」とは異なり、LLMが動的に自身のプロセスやツールの使用を指示し、タスクをどのように達成するかを制御する。

#### 各社の定義

- **Google Cloud:** ユーザーの代わりに目標を追求し、タスクを完了させるソフトウェアシステム。推論・計画・記憶が可能で、意思決定や学習、適応を行う自律性を備える。
- **Anthropic:** LLMが動的に自身のプロセスやツールの使用を指示するシステム。「ループ内でツールを使用するLLM」と表現される。
- **OpenAI:** ユーザーの代わりに独立して仕事を行うことができるAIシステム。

### 構成要素

| 構成要素 | 役割 | 具体例 |
| :--- | :--- | :--- |
| **モデル(頭脳)** | 言語の理解、推論、計画の立案、行動の決定 | Gemini、Claude、GPT-4など |
| **ツール(手足)** | データ取得、計算、外部システム操作 | Web検索、API呼び出し、コード実行 |
| **メモリ(記憶)** | 短期記憶(現在のコンテキスト)と長期記憶(過去の対話や経験)を保持 | 対話履歴、ベクトルDB、知識ベース |
| **計画(段取り)** | 複雑なタスクを管理可能なステップに分解し、最適な行動方針を選択 | OODAループ、ReActフレームワーク |
| **ペルソナ(役割)** | システムプロンプトを通じて役割や行動指針を定義し、一貫した振る舞いを保証 | システムプロンプト、ジョブディスクリプション |

人間に例えると、モデルは脳、ツールは手足やパソコン、メモリは記憶やノート、計画は段取り力、ペルソナは職務記述書にあたります。

## 1. 前提認識:社内における「エージェント」の定義と現状

「エージェント」という言葉が社内で複数の意味で使われている現状を共有し、本日の議論のスコープを明確化。

### 定義(2つのスタンス)

社内における「AIエージェント」の定義は、**「構造(どう作られているか)」**を重視する立場と、**「自律性(どう振る舞うか)」**を重視する立場で分かれています。

- **広義の定義(生成AIプロジェクト:構造重視)**
  - **定義:** エージェント・フレームワークを用いて構築された機能全般。  
  特定のタスクを自動化・支援する **「ワークフロー型」や「アシスタント機能」** も含む広い概念。
  - **判断基準:** AIが単に回答するだけでなく、検索・計算・ファイル操作などの**「ツール(道具)」を組み合わせて一連の工程を実行する仕組み**であればエージェントと呼称。
  - **範囲:** 自律型(GPS業務サポート等)、定型・支援型(PowerPoint Agent、育休コンシェルジュ等)、簡易作成型(Workspace Studio)を含む。
- **狭義の定義(GPS・EOL・CTO:自律性重視)**
  - **GPS(荒木氏):** AIが主体的に考え、学習し、行動する **「自律型AI(Autonomous AI)」**。単なる「アシスタント」や「ボット」とは明確に区別すべきという立場。
  - **EOL(三好CTO):** BigTech(Google等)のLLMに依存せず、学習可能なLLMを使って構築する荏原独自の **「自律分散型AIエージェント」**。広義のエージェント(BigTech依存・ツール利用型)と用語が混同されないよう整理を要請。

### 構成要素(エコシステム)

生成AIプロジェクトが定義する「広義のエージェント」は、以下の要素を組み合わせた **「エコシステム」** として構成。

| 業界標準の構成要素 | エコシステム要素 | 役割 | 具体例 |
| :--- | :--- | :--- | :--- |
| **モデル(頭脳)** | **Models** | 推論や判断を行うLLM | AI API(Python/GAS)<br> |
| **ツール(手足)** | **Tools/MCPs** | AIが実行する具体的機能や外部接続 | Web検索<br>ファイル操作<br>社内システム接続 |
| **ツール(手足)** | **APIs** | 他システムやデータを呼び出す接続口 | Salesforce API<br>Gmail API |
| **メモリ(記憶)** | **Data** | AIが参照・加工する情報源 | 非構造化データ(文書、画像、動画)<br>構造化データ(DB、Excel) |
| **計画(段取り)** | **Agents** | 特定目的を持つAI機能の塊 | なし |
| **ペルソナ(役割)** | **UI/Apps** | ユーザーとの接点 | プロンプト<br>Dify<br>AgentSpace(EBARA AI Chat) |

### 今回のスコープ

本日の議論では、**生成AIプロジェクトが定義する「広義のエージェント(自動化・支援ツール)」**、特にEBARA AI Chatの新機能や「Workspace Studio」で利用可能な業務効率化機能を対象。

### 業界標準と社内定義の違い

業界標準の定義と社内定義には大きなギャップが存在。

| 項目 | 業界標準(Google・Anthropic・OpenAI) | 社内定義(生成AIプロジェクト) |
| :--- | :--- | :--- |
| **核心要件** | LLMが**自律的にループ**して目標達成 | **ツール利用**があれば広くエージェントと呼称 |
| **自律性** | 動的な計画立案・修正が必須 | 事前定義済みワークフローも含む |
| **複雑性** | 不確実な状況での試行錯誤を想定 | 定型・支援業務も対象範囲 |
| **判断基準** | 「考えて行動する」システム | 「道具を使う」システム |

**率直な見解:** 社内では「エージェント」という響きの良さから広義に定義しているが、現在提供している機能の多くは業界標準でいう「ワークフロー」や「アシスタント」に近く、真の意味でのエージェントは限定的。ただし、ユーザーの業務効率化には一定の効果。

## 2. 社内におけるエージェントの強み・弱み

生成AIプロジェクトの考える強み・弱みを提示。  
従来の「RPA」や「単発の生成AI(チャット)」と比較して整理。

- **メリット・強み(自律性と連続性)**
  - **目標達成能力:** 「〇〇を翻訳して」といった指示に対し、自ら計画し行動する。
  - **連続性:** 一問一答ではなく、複数のステップ(検索→要約→メール作成など)を連続して実行できる。
  - **ツール連携:** 社内データ、Web検索、カレンダー、メールなどを横断して利用可能。
- **デメリット・弱み(不確実性と責任)**
  - **不確実性:** 自律的に動くため、毎回全く同じ挙動をするとは限らない(RPAとの違い)。
  - **再現性の低さ:** モデルのアップデートや参照データの変化により、結果が変わる可能性がある。
  - **ガバナンス:** 「どこまでAIに任せるか(承認は人間がやるか)」という**HITL(Human-in-the-Loop)**の設計が必要。

## 3. 現段階の社内「エージェント」ラインナップ

現在利用可能、または開発中の具体的なエージェントを提示し、イメージを共有。

- **① 単作業領域のAI活用事例**
  - **EOL Ideation:** 研究領域の課題分析とアイデア出し。
  - **PowerPoint Agent:** パワポ資料をフォーマットを崩さず翻訳(ベータ版提供中)。
  - **図面チェックリスト自動化(エネルギー):** Geminiを活用し、図書レビューとリスト作成を自動化。
  - **図面からの部品番号抽出(精密):** 改造指示書から変更パーツ番号を抽出し検索可能に。
  - **不安全行動検知(環境・EEP):** 動画を解析し、危険な作業行動を検知。
  - *含む要素：モデル、ペルソナ / 含まない要素：ツール、計画、メモリ*
- **② 特定業務特化型(事業部連携)**
  - **GPS業務サポート:** 見積書を読み込み、過去データと比較して妥当性を査定・解説。
  - **育休コンシェルジュ:** 規定に基づいた回答＋カレンダー登録支援。
  - *含む要素：モデル、ツール、ペルソナ / 含まない要素：計画、メモリ*
- **③ 全社標準・汎用ツール(プロジェクト提供)**
  - **Workspace Studio (旧Flows):** Google Workspace(Gmail, Drive等)と連携し、ノーコードで簡易エージェントを自作できるツール。
  - **Dify:** エージェント・アシスタント・ワークフローを構築できるノーコード開発プラットフォーム。
  - *含む要素：モデル、ツール、ペルソナ、計画 / 含まない要素：メモリ*

## 4. 統括部業務・生産領域での活用ディスカッション

上記を踏まえ、具体的にどこで使えそうか(Where)を議論。

### A. 統括部業務での活用可能性

- **① 単作業領域のAI活用**
  - 技術文書の翻訳・要約、図面レビューのチェックリスト作成、安全動画の自動解析など
- **② 特定業務特化型**
  - 統括部向けカスタムエージェント（予算査定支援、規程問い合わせ対応など）の開発可能性
- **③ 全社標準・汎用ツール**
  - メールの自動振り分け・要約、カレンダー調整、ドライブ内のファイル整理など定型業務の自動化
  - *誰でも作れる「簡易エージェント」としてトライアルしやすい領域*

### B. 生産領域での活用可能性

- **① 単作業領域のAI活用**
  - 図面・仕様書・検査成績書の読み取り・照合・リスト化（エネルギー・精密の事例展開）
  - 動画や画像を用いた不安全行動検知や外観検査支援（環境の事例展開）
- **② 特定業務特化型**
  - 生産現場向け専門エージェント（設備診断支援、品質管理アシスタントなど）のカスタム開発
- **③ 全社標準・汎用ツール**
  - 実験データやExcelデータの前処理・加工（データサイエンスGと連携した前処理エージェントの構想）
  - 生産計画や進捗管理の自動化支援

## 補足:認識合わせのポイント(アナロジー)

エージェント開発を**「料理」**に例えて整理。

- **生成AI(チャット):** 冷蔵庫にある食材(データ)について質問すると、レシピを教えてくれるアドバイザー。
- **エージェント:** レシピを考えるだけでなく、包丁や鍋(ツール)を自分で使い、食材を切って煮込んで料理(成果物)まで作ってくれる**見習いシェフ**。
  - ただし、「見習い」なので、味付けが毎回微妙に違ったり(不確実性)、火加減を間違えることもある。
  - だからこそ、最後にお客様(次の工程)に出す前に、**料理長(人間)の味見(確認・承認)**が必要です。

この「味見(Human-in-the-Loop)」をどの工程に挟むかが、生産領域での活用における重要なポイント。
